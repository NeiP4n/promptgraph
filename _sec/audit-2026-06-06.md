### 1. Dependency: `tar` — 6 known high-severity CVEs (arbitrary file write, symlink traversal, race condition)

  | GHSA-8qq5-rm4j-mr97 | — | — | Arbitrary file overwrite via insufficient path sanitization |

### 2. Rate Limiter — TOCTOU race condition allows burst bypass

- **Rule**: `sec_dangerous_patterns` → `js-race-condition`
- **CWE**: CWE-367 (TOCTOU Race Condition)
- **Location**: `src/utils/rate-limiter.js:9-11`
- **Evidence**:
  ```js
  async acquire() {
    while (!this.tryAcquire()) {
      await new Promise(r => setTimeout(r, Math.min(this.windowMs, 100)))
    }
  }

  tryAcquire() {
    const now = Date.now()
    const cutoff = now - this.windowMs
    this.timestamps = this.timestamps.filter(t => t > cutoff)
    if (this.timestamps.length >= this.maxRequests) return false
    this.timestamps.push(now)
    return true
  }
  ```
- **Why it's exploitable**: The `acquire()` method has a classic TOCTOU flaw. Between the `while` check and the `await setTimeout()`, the async context yields to the event loop. If N concurrent requests call `acquire()` simultaneously, they ALL can pass `tryAcquire()` (which filters + checks + pushes synchronously) before any of them returns. The `async acquire()` + `await setTimeout()` pattern is specifically vulnerable because multiple coroutines all enter the `while` loop concurrently, pass `tryAcquire()`, and exceed `maxRequests`. This defeats the entire purpose of rate limiting.
- **Fix**: Replace the spin-loop pattern with a proper queue-based approach using a token bucket or sliding window with a mutex (`better-queue`, `token-bucket` npm package, or use `setInterval`-based token refill). Example:
  ```js
  tryAcquire() {
    const now = Date.now()
    const cutoff = now - this.windowMs
    this.timestamps = this.timestamps.filter(t => t > cutoff)
    if (this.timestamps.length >= this.maxRequests) return false
    this.timestamps.push(now)
    return true
  }
  // Remove acquire() — use tryAcquire() directly in a synchronous context,
  // or wrap with a proper Promise queue.
  ```

## Medium Severity

### 3. Content-Length bypass via UTF-8 multi-byte chunk splitting

- **Rule**: `sec_dangerous_patterns` → `js-unsafe-buffer`
- **CWE**: CWE-770 (Allocation of Resources Without Limits or Throttling)
- **Location**: `github-import.js:28-57` (`streamDownload`)
- **Evidence**:
  ```js
  const cl = parseInt(r.headers['content-length'], 10)
  if (!isNaN(cl) && cl > maxSize) {         // ← initial check: server-supplied header
    r.resume()
    return rej(new Error(`Content-Length ${cl} exceeds max ${maxSize}`))
  }
  r.setEncoding('utf8')
  r.on('data', c => {
    total += Buffer.byteLength(c, 'utf8')   // ← byte counting
    if (total > maxSize) { ... }
    d += c
  })
  ```
- **Why it's exploitable**: The Content-Length header check is a fast pre-filter, but the real defense is the byte counter. However, `Buffer.byteLength(c, 'utf8')` when `c` is already a UTF-8 decoded string (via `r.setEncoding('utf8')`) counts the **decoded** character byte length, not the original bytes. If an attacker-controlled server sends content where multi-byte UTF-8 sequences (e.g., 4-byte emoji) are split across TCP chunks, the stream decoder may emit replacement characters (`\uFFFD`), each counting as 3 bytes in `Buffer.byteLength`. A carefully crafted sequence of split multi-byte sequences could cause the byte counter to undercount by up to ~25%, allowing more data through than `MAX_DOWNLOAD_SIZE` permits. Additionally, `d += c` in a loop is O(n²) for very large payloads, enabling a CPU-exhaustion DoS.
- **Fix**: Count raw bytes from the response stream (before UTF-8 decoding) by tracking `r.headers['content-length']` and using integer comparison on raw byte chunks. Replace `r.setEncoding('utf8')` with `r.on('data', chunk => { total += chunk.length; ... })` and decode only after validation. Replace `d += c` with array push+join pattern.

### 4. Global state race condition in search.js — `setHybridWeights`

- **Rule**: `sec_dangerous_patterns` → `js-global-mutation`
- **CWE**: CWE-362 (Concurrent Execution using Shared Resource with Improper Synchronization)
- **Location**: `search.js:5-11`, `search.js:13-23`, `search.js:119-125`
- **Evidence**:
  ```js
  let embedWeight = 0.7   // module-level mutable global
  let bm25Weight = 0.3

  export function setHybridWeights(ew, bw) {
    embedWeight = ew
    bm25Weight = bw
  }

  function adaptWeights(query) {
    if (hasTechTerms) {
      embedWeight = 0.5     // ← mutates global during search!
      bm25Weight = 0.5
    } else {
      embedWeight = 0.7
      bm25Weight = 0.3
    }
  }

  // used 80 lines later in the same search():
  score: embedWeight * s.embedScore + bm25Weight * s.bm25Score
  ```
- **Why it's exploitable**: The MCP server runs concurrent requests. If request A calls `search('API query')` which sets `embedWeight = 0.5` and then yields to the event loop, request B could call `search('design ideas')` which sets `embedWeight = 0.7` AGAIN. When request A resumes, it uses the overwritten weights, producing incorrect ranking. More critically, if `setHybridWeights(0.0, 1.0)` is called by one request while another is in-flight, the in-flight search returns corrupted scores (BM25-only or embedding-only), potentially leaking information or causing denial of service.
- **Fix**: Make weights local to each `search()` call — do not mutate module-level globals. Either pass weights as a parameter through the call chain, or compute them inline without side effects:
  ```js
  export async function search(query, topK = 5) {
    const [embedWeight, bm25Weight] = computeWeights(query)
    // ... use local consts
  }
  ```

### 5. `sanitizeExternalContent` crashes on null/undefined input + dead code in production

- **Rule**: `sec_dangerous_patterns` → `js-unvalidated-input`
- **CWE**: CWE-754 (Improper Check for Unusual or Exceptional Conditions)
- **Location**: `validator.js:160-168`
- **Evidence**:
  ```js
  export function sanitizeExternalContent(content) {
    let sanitized = content.replace(/\0/g, '')  // ← TypeError if content is null/undefined
    if (Buffer.byteLength(sanitized, 'utf8') > MAX_DOWNLOAD_SIZE) {
      sanitized = Buffer.from(sanitized, 'utf8').subarray(0, MAX_DOWNLOAD_SIZE).toString('utf8')
    }
    return sanitized
  }
  ```
- **Why it's exploitable**: No guard against `null`/`undefined`. If this function is ever called with external content that is null (e.g., a failed fetch, empty API response), it throws `TypeError: Cannot read properties of null`. **Additionally**, this function is **exported but never imported by any production module** — it's only called from tests. It is dead code. If it's intended for future use, it needs fixing before activation.
- **Fix**: Add null/undefined guard:
  ```js
  export function sanitizeExternalContent(content) {
    if (content == null) return ''
    // ...
  }
  ```
  This also needs to be integrated into the download pipeline (e.g., in `streamDownload` or `installSkill`).

### 6. Rate limiter memory leak — unbounded timestamp array

- **Rule**: `sec_dangerous_patterns` → `js-memory-leak`
- **CWE**: CWE-770 (Allocation of Resources Without Limits or Throttling)
- **Location**: `src/utils/rate-limiter.js:5-6, 17`
- **Evidence**:
  ```js
  constructor({ maxRequests, windowMs }) {
    this.maxRequests = maxRequests
    this.windowMs = windowMs
    this.timestamps = []     // ← grows unbounded
  }

  tryAcquire() {
    const now = Date.now()
    const cutoff = now - this.windowMs
    this.timestamps = this.timestamps.filter(t => t > cutoff)  // filters but never compacts
    if (this.timestamps.length >= this.maxRequests) return false
    this.timestamps.push(now)
    return true
  }
  ```
- **Why it's exploitable**: Under sustained near-capacity load, the timestamps array holds up to `maxRequests` entries (default 30), which is fine. But during bursts, it can temporarily exceed this. More critically, no upper bound is enforced — if `tryAcquire()` is called millions of times, every call does a full array filter (O(n) scan). A sustained attack of just-below-rate-limit requests causes O(n) filtering on every call. With the GitHub import limiter (30 req/min), this is not immediately dangerous, but it wastes CPU on every request.
- **Fix**: Implement a circular buffer or fixed-size sliding window:
  ```js
  constructor({ maxRequests, windowMs }) {
    this.maxRequests = maxRequests
    this.windowMs = windowMs
    this.timestamps = new Array(maxRequests).fill(0)
    this.idx = 0
  }

  tryAcquire() {
    const now = Date.now()
    const cutoff = now - this.windowMs
    const oldest = this.timestamps[this.idx]
    if (oldest > cutoff) return false  // window full
    this.timestamps[this.idx] = now
    this.idx = (this.idx + 1) % this.maxRequests
    return true
  }
  ```

### 7. Prototype pollution via registry entry spread operator

- **Rule**: `sec_dangerous_patterns` → `js-prototype-pollution`
- **CWE**: CWE-1321 (Prototype Pollution)
- **Location**: `marketplace.js:175`, `marketplace.js:300`
- **Evidence**:
  ```js
  // marketplace.js:175
  return registry.skills
    .map(s => ({ ...s, code: s.code || codeFor(s.id) }))  // ← spread of untrusted data

  // marketplace.js:300
  .map(b => ({ ...b, code: b.code || codeFor(b.id) }))
  ```
- **Why it's exploitable**: The `{ ...s }` spread operator on registry entries from `raw.githubusercontent.com` could introduce `__proto__`, `constructor`, or `prototype` properties if the registry JSON is compromised (e.g., via a malicious registry update or MITM). While a basic `{ ...__proto__ }` doesn't pollute the prototype (spread creates own properties only), nested processing of these objects later could trigger pollution. A malicious registry entry with `{ "__proto__": { "isAdmin": true } }` spread into a new object becomes `{ isAdmin: true }` — not prototype pollution per se, but the property leaks into the result object where it might be read by downstream code.
- **Fix**: Validate/filter registry entries before spreading. Use `Object.keys(s).filter(k => !k.startsWith('__'))` or explicitly pick known fields:
  ```js
  const SAFE_KEYS = ['id', 'name', 'description', 'version', 'author', /* ... */]
  const safe = {}
  for (const k of SAFE_KEYS) if (k in s) safe[k] = s[k]
  ```

## Low Severity

### 8. No access control on `setTrustLevel` / `rateSkill` — internal functions

- **Rule**: `sec_dangerous_patterns` → `js-missing-auth`
- **CWE**: CWE-862 (Missing Authorization)
- **Location**: `marketplace.js:569-581`, `marketplace.js:611-631`
- **Evidence**:
  ```js
  export async function setTrustLevel(name, level) { ... }   // no auth check
  export async function rateSkill(name, rating) { ... }      // no auth check
  ```
- **Why it's exploitable**: These functions accept a skill `name` and modify the database. Though they're not currently exposed via MCP tools, any internal module can call them. If a future MCP tool exposes them, any LLM/MCP client could set any skill's trust level to "verified" or manipulate ratings.
- **Fix**: Add authorization check (e.g., require `GITHUB_TOKEN` or registry API key). Or scope trust levels to only be manageable by the registry itself.

### 9. `spawnSync` with `shell: true` in update path

- **Rule**: `sec_dangerous_patterns` → `js-shell-injection`
- **CWE**: CWE-78 (OS Command Injection)
- **Location**: `index.js:654`
- **Evidence**:
  ```js
  const result = spawnSync('npm', ['install', '-g', 'promptgraph-mcp@latest'], {
    encoding: 'utf8', stdio: 'pipe', shell: true    // ← shell: true
  })
  ```
- **Why it's exploitable**: The version string comes from npm registry API response, so it's semver-safe today. But `shell: true` passes the entire command through the shell (`cmd.exe` on Windows). If the `latest` version from the registry ever contained shell metacharacters (e.g., `2.4.9 & echo pwned`), `shell: true` would execute them. Though the current value is safe, this is a defense-in-depth violation.
- **Fix**: Remove `shell: true` or use it only when necessary. The args array is already split — `shell: true` is redundant for non-Windows and dangerous on Windows where `cmd /c` wraps the command.

### 10. Regex performance risk in reranker — `queryTerms.forEach(t => text.includes(t))`

- **Rule**: `sec_dangerous_patterns` → `js-regex-dos`
- **CWE**: CWE-1333 (Inefficient Regular Expression Complexity)
- **Location**: `src/reranker/reranker.js:17`
- **Evidence**:
  ```js
  const queryTerms = (query.toLowerCase().match(/\w+/g) || [])
  // ...
  const termOverlap = queryTerms.filter(t => text.includes(t)).length / queryTermCount
  ```
- **Why it's exploitable**: For a query with many terms (e.g., 1000 word-chars), each term triggers `text.includes(t)` on every result's `text` field. If results have large text content (400 chars each from `skillWithSnippet`), and there are 20 results (the `topN` in search.js:131), this means 20 × 1000 = 20,000 `includes()` calls per search. While `includes()` is fast (native), an attacker could craft a very long query string with thousands of `\w+` sequences to cause CPU-bound search delays.
- **Fix**: Limit query term count (e.g., `queryTerms.slice(0, 50)`) and/or restrict `results` array size before reranking.

### 11. Embedder hard limit at 10000 chunks — unrecoverable after exhaustion

- **Rule**: `sec_dangerous_patterns` → `js-hard-limit`
- **CWE**: CWE-770 (Allocation of Resources Without Limits or Throttling)
- **Location**: `embedder.js:36-38`
- **Evidence**:
  ```js
  if (embedCallCount + texts.length > MAX_EMBEDDING_CALLS) {
    throw new Error(`Embedding queue limit exceeded (max ${MAX_EMBEDDING_CALLS} chunks per session)...`)
  }
  ```
- **Why it's exploitable**: After 10,000 chunks are embedded in a single process lifetime, ALL embedding operations permanently fail with no reset mechanism. `resetEmbedCallCount()` exists but is not called in any production flow. A series of large reindex operations could exhaust this limit and render the server non-functional until restart.
- **Fix**: Set limit higher (100000) or auto-reset every hour / per reindex cycle.

3. **[MEDIUM]** Make `search.js` weights local to each call instead of shared mutable globals — pass `[embedWeight, bm25Weight]` through the call chain.
4. **[MEDIUM]** Fix `streamDownload` to count raw bytes before UTF-8 decoding, preventing the undercount vector.
5. **[MEDIUM]** Add null guard to `sanitizeExternalContent` and integrate it into the actual download pipeline.
6. **[MEDIUM]** Optimize rate limiter timestamp storage to circular buffer (prevent memory leak / O(n) filtering).
7. **[LOW]** Add field whitelist when spreading registry entries to prevent prototype property leakage.
8. **[LOW]** Remove `shell: true` from `spawnSync` in update path.
9. **[LOW]** Add term limit in reranker (max 50 query terms) to prevent CPU exhaustion.
